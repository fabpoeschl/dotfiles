Debug Configurations ~

  Python:
    Launch file                         Run current .py file
    Launch with arguments               Prompts for CLI arguments
    Attach remote (localhost:5678)      For kubectl port-forward / debugpy
    Attach remote (custom host:port)    Prompts for host and port

  JavaScript / TypeScript:
    Launch file (Node)                  Run current file with Node
    Attach to Node (localhost:9229)     For --inspect / kubectl port-forward
    Attach to Node (custom host:port)   Prompts for host and port
    Launch Chrome                       Debug in Chrome (prompts for URL)

  Ruby:
    Launch file                         Run current .rb file
    Launch with bundler                 bundle exec ruby <file>
    Launch Rails server                 bundle exec rails server
    Launch RSpec (current file)         bundle exec rspec <file>
    Attach remote (localhost:1234)      For rdbg / kubectl port-forward
    Attach remote (custom host:port)    Prompts for host and port

Remote Debugging (Kubernetes) ~

  The "Attach remote" configurations connect to a debug server over TCP.
  Use kubectl port-forward to bridge the pod's debug port to localhost:

    # Python (debugpy)
    kubectl port-forward pod/my-app 5678:5678
    # Then: <F5> -> "Attach remote (localhost:5678)"

    # JavaScript / Node (--inspect)
    kubectl port-forward pod/my-app 9229:9229
    # Then: <F5> -> "Attach to Node (localhost:9229)"

    # Ruby (rdbg / debug.gem)
    kubectl port-forward pod/my-app 1234:1234
    # Then: <F5> -> "Attach remote (localhost:1234)"

  On the pod, start the app with its debug server enabled:

    # Python:  python -m debugpy --listen 0.0.0.0:5678 --wait-for-client app.py
    # Node:    node --inspect=0.0.0.0:9229 app.js
    # Ruby:    rdbg --open --host 0.0.0.0 --port 1234 -c -- ruby app.rb

PER-PROJECT CONFIGURATION                          *cheatsheet-project-config*

  exrc is enabled globally (see options.lua), so a .nvim.lua at the
  project root is auto-sourced from the cwd. Run :trust on first open
  to authorize it. CLI flags always win over these defaults.

  Single-default fields ~

    -- .nvim.lua
    vim.g.k8s_context             = "<your-context>"     -- DBConnect / PodConnect
    vim.g.k8s_namespace           = "<your-namespace>"   -- DBConnect / PodConnect
    vim.g.dbconnect_database      = "<workload>"         -- :DBConnect workload
    vim.g.dbconnect_user          = "<db-user>"          -- :DBConnect user
    vim.g.dbconnect_credentials_cmd = "scripts/db-credentials.sh"
    vim.g.podconnect_application  = "<app-workload>"     -- :PodConnect default
    vim.g.dap_remote_root         = "/srv/app"           -- DAP pathMappings

  Presets (multiple environments / databases) ~

    For projects with several context+workload combinations, define a
    presets table. Pass the preset name as a positional arg, or run
    :DBConnect / :PodConnect bare to pick from a vim.ui.select. Tab-
    completion suggests preset names.

    vim.g.dbconnect_presets = {
      ["staging/core"]    = { context = "<staging-ctx>", namespace = "<staging-ns>",
                              database = "<staging-core-workload>",   local_port = "5432" },
      ["staging/metric"]  = { context = "<staging-ctx>", namespace = "<staging-ns>",
                              database = "<staging-metric-workload>", local_port = "5433" },
      ["prod/core"]       = { context = "<prod-ctx>",    namespace = "<prod-ns>",
                              database = "<prod-core-workload>",      local_port = "5434" },
      ["prod/metric"]     = { context = "<prod-ctx>",    namespace = "<prod-ns>",
                              database = "<prod-metric-workload>",    local_port = "5435" },
    }

    vim.g.podconnect_presets = {
      ["staging/api"] = { context = "<staging-ctx>", namespace = "<staging-ns>", app = "<api-workload>" },
      ["prod/api"]    = { context = "<prod-ctx>",    namespace = "<prod-ns>",    app = "<api-workload>" },
    }

    Recognized preset keys mirror the CLI flags:
      DBConnect: context, namespace, database, user, port, local_port, dbname
      PodConnect: context, namespace, app

    Distinct local_port per preset is what lets multiple connections
    coexist (otherwise both would try to bind localhost:5432).

  Splitting presets across files ~

    .nvim.lua is just lua, so you can keep contexts in separate files
    and aggregate them:

      -- .nvim.lua
      vim.g.dbconnect_presets = {}
      vim.g.podconnect_presets = {}
      for _, f in ipairs(vim.fn.glob(".nvim.d/*.lua", false, true)) do
        dofile(f)
      end

      -- .nvim.d/lab.lua
      vim.g.dbconnect_presets["lab/core"]   = { ... }
      vim.g.dbconnect_presets["lab/metric"] = { ... }

    :trust only covers .nvim.lua; the dofile'd files run unsandboxed,
    so only use this for files you wrote yourself.

  Machine-local config (no .nvim.lua in the repo) ~

    Per-project config can also live in ~/.config/nvim/projects/<name>.lua.
    These files are excluded from the dotfiles repo (.gitignore /
    .chezmoiignore) so they never get committed anywhere. This is useful
    when the project repo has no .nvim.lua, or when you don't want to
    maintain one per git-worktree.

    The project name is the basename of the git main-worktree root, derived
    from `git rev-parse --git-common-dir`. The same file is therefore used
    across all linked worktrees of the same repository. To find the name
    Neovim will look for, run inside your project:

      git rev-parse --git-common-dir | xargs dirname | xargs basename

    Create the file at that path with the same vim.g variables as above:

      -- ~/.config/nvim/projects/my-app.lua
      vim.g.k8s_context  = "my-cluster"
      vim.g.k8s_namespace = "production"
      vim.g.dbconnect_presets = { ... }

    No :trust prompt is required since the file is in your own config dir.
    A .nvim.lua in the worktree directory takes precedence (exrc runs first).

  Resolution order ~

    1. CLI flag passed to the command
    2. preset values (if a preset name was given or chosen via picker)
    3. vim.g.<name> (single-default field)
    4. environment variable (where applicable, e.g. $DBCONNECT_USER)
    5. interactive prompt or hardcoded fallback

DATABASE (vim-dadbod)                                *cheatsheet-database*

Connection Management ~
  <leader>Du        Toggle DBUI drawer
  <leader>Da        Add a connection manually
  <leader>Dc        Connect to remote DB (K8s pod)
  <leader>Dd        Disconnect remote DB

Remote Connect (:DBConnect) ~
  :DBConnect                                  Pick from preset list (if any)
  :DBConnect <preset>                         Use a preset from .nvim.lua
  :DBConnect <preset> -u readonly             Preset + override one field
  :DBConnect -c <ctx> -n <ns> -d <workload>   Pure flags
  :DBConnect -c <ctx> -n <ns> -d <workload> -p 5433 -l 15432 -b mydb -u admin

  See PER-PROJECT CONFIGURATION for the preset table format.

  Flags:
    -c, --context     Kubernetes context
    -n, --namespace   Kubernetes namespace
    -d, --database    Deployment / StatefulSet name (resolved via ownerRefs)
    -u, --user        DB user (default: $DBCONNECT_USER or "postgres")
    -p, --port        Remote port (auto-detected from name if omitted)
    -l, --local-port  Local port to forward to (default: same as remote)
    -b, --dbname      Database name in URL (defaults to -d value)

  :DBDisconnect [name]  Stop port-forward and remove connection
                        Tab-completes active connection names; with no
                        argument prompts via vim.ui.select.

Credentials ~
  Set $DBCONNECT_CREDENTIALS_CMD to a script that prints either:
    password                   (1 line, uses default user)
    username\npassword         (2 lines, supplies both)
  The script receives -n <ns> -d <database> -c <context>.
  If unset, :DBConnect prompts via inputsecret().

Local Databases ~
  Use <leader>Da to add a local connection URL directly:
    postgres://user:pass@localhost:5432/mydb
    mysql://root@localhost:3306/mydb
    sqlite:path/to/db.sqlite

KUBERNETES                                        *cheatsheet-kubernetes*

Pod Connection (:PodConnect) ~
  <leader>ks        Shell into pod (prompts for context/ns/app)
  <leader>kl        Tail pod logs
  <leader>kf        Port-forward pod

  :PodConnect                                            Pick preset (if any)
  :PodConnect <preset>                                   Preset, action=shell
  :PodConnect <preset> logs                              Preset + action
  :PodConnect -c <ctx> -n <ns> -a <app>                  Shell (default)
  :PodConnect -c <ctx> -n <ns> -a <app> logs             Tail logs
  :PodConnect -c <ctx> -n <ns> -a <app> forward 3000:3000

  All actions open in a terminal split. Close the buffer to stop.
  If the first positional matches a preset name it is consumed as a
  preset; otherwise it is treated as the action.

Pod matching ~
  -a <app> resolves to a Deployment / StatefulSet / DaemonSet / Job by
  inspecting each pod's ownerReferences (for ReplicaSet owners the trailing
  pod-template-hash is stripped to get the Deployment name).

Shell Scripts (~/.local/bin/) ~
  remote-db-connect -c <ctx> -n <ns> -d <workload> [-p <port>] [-l <local-port>]
  remote-pod-connect -c <ctx> -n <ns> -a <workload> [shell|forward|logs]
